# DBUF Protocol

An application layer protocol defining structures for requests, responses, stream management and authentication.

## Transport Layer

For internet wide connections, the standard transport layer is QUIC on port 443. The TLS ALPN value for current research and prototyping is "dbuf/demo". Another ALPN value will be published when core semantics are settled.

## Stream Format

A protocol stream is a series of DBUF items encoded with the basic encoding.

A stream optionally begins with a stream group identifier. If the first item's basic type is unsigned integer, it is interpreted as the stream group identifier. If the stream group identifier is omitted it is implied to be zero. Stream group semantics will be defined in a future specification.

Next, a stream consists of a series of payloads. A payload consists of headers, an optional body and optional footers.

Headers are encoded as a DBUF map. The body may be embedded in the headers using [value](./registry/value.md) as the map key. If [body_length](./registry/body_length.md) is present in the headers, it expresses the byte length of the body or [describe_no_value](./registry/describe_no_value.md) indicates the body will end with a [end_marker](./registry/end_marker.md) item. The body may consist of multiple DBUF items.

If [body_length](./registry/body_length.md) is present in the headers, the footers are represented by one DBUF item after the body.

## Header Store

The receiving side of a connection can store data to be reused in headers across streams and payloads.

[header_store](./registry/header_store.md) has the following semantics when present in headers:

- If value is an unsigned integer, specifies the id of store data to be used in this payload. If id has not been previously set, return an error. If id exists, merge stored headers with payload headers. If [reference](./registry/reference.md) exists in both stored and payload headers, append payload reference to stored reference. If any other headers exist in both stored and payload headers, return an error.
- If value is an array, the first item is an unsigned integer specifying the id of the store data. The second item is a map of headers to be stored. If the array contains more than two items, the third item is an [identity](./registry/identity.md) id and the fourth item is an [identity_key](./registry/identity_key.md) id. Any additional items are parameters for authentication according to the configured algorithm of the [identity_key](./registry/identity_key.md)


## Addressing

A globally resolvable [reference](./registry/reference.md) consists of an array of DBUF items. Registry symbols provide standardized elements within the array.

When the first item is [host](./registry/host.md) and the second item is type text, the second item is interpreted as a DNS name. If [port](./registry/port.md) is the third item, an unsigned integer in the fourth item is interpreted as the port number.

[authority_marker](./registry/authority_marker.md) separates the authority and subpath portions of a reference. Any items between the host/port and [authority_marker](./registry/authority_marker.md) designate a sub authority.

A [reference](./registry/reference.md) is included in the headers of a payload for routing purposes.

## Errors

If an error occurs in a request before sending response headers, the error is encoded in an [error](./registry/error.md) header and the response will not have a body.

If an error occurs after a response body has begun streaming, the stream is cancelled and the numeric value of [error](./registry/error.md) is signalled through the transport layer. Then the full error is sent in the [error](./registry/error.md) header on a separate stream. The headers of the separate stream also contain [reference](./registry/reference.md) with a value of [error](./registry/error.md) and [stream_id](./registry/stream_id.md) with the id of the original stream.

## Identity

A new identity is created by sending a request to the [identity](./registry/identity.md) subpath of an authority.

The body of the request is a map with [identity_key](./registry/identity_key.md) containing an inner map where the keys are user supplied ids for authentication keys to be configured.

The value of each inner key is an array with the first item specifying the type of authentication key and the remaining items are parameters for that key type.

The response contains the identity id generated by the server.

## Authentication

An identity authenticates by providing a digital signature and context data as part of the header store parameters of a request. The authenticated identity can be reused on subsequent requests merely by specifying the header store id.

The message to be signed is the concatenation of:

- The string "DBUF demo transport authentication" in UTF8 encoding.
- Exporter context
- The first 32 bytes of exporter output.


Exporter context is the concatenation of the DBUF item encodings of:

- The header store headers.
- The [identity](./registry/identity.md) id
- The [identity_key](./registry/identity_key.md) id
- The array of parameters configured on creation of the [identity_key](./registry/identity_key.md)


Exporter output is the result of the export keying material function of the transport layer. For TLS refer to Exporters in RFC 8446. The parameter of the export function are:

- "EXPORTER: DBUF demo transport authentication"
- The exporter context defined above.


The header store parameters are the signature and the 33rd-48th bytes of the exporter output.

To validate the authentication on the receiving end, reconstruct the exporter context, exporter output and signing message, then validate the signature and validate the 33rd-48th bytes of the exporter output match the header store parameter.

The protected headers must contain a [reference](./registry/reference.md) with an authority for the corresponding server.
